十 系统级I/O
  • 序言
    • IO是主存和外部设备之间复制数据的过程
    • linux中用系统级IO函数实现较高级别的
  • 10.1 系统级IO
    • 原理
      • 所有的IO设备（网络，磁盘，终端）都被模型化为文件
      • 文件就是一个字节的序列
      • 所有的IO操作都被当前当作相应的文件读写
    • 所有输入输出的过程
      • 1.打开文件
        • 打开文件，应用程序要求内核打开文件，返回一个小的非负整数描述符
        • 描述符标识该文件
        • 内核记录有关打开这个文件的所有信息
        • 应用程序只需要记住描述符
      • 每个进程开启时的三个打开的文件
        • 标准输入，描述符0
        • 标准输出，描述符1
        • 标准错误。描述符2
      • 2.改变当前的文件位置
        • 文件位置：从文件开头开始的字节偏移量
      • 3.读写文件
        • 读，就是从文件复制若干字节到内存
        • 读结束后，触法EOF条件
        • 在文件结尾处并没有明确的EOF符号
      • 4.关闭文件
        • 内核释放文件打开时创建的数据结构，把描述符恢复到描述符池中
        • 无论进程因为什么原因终止，内核都会关闭所有打开的文件，并释放内存
  • 10.2文件
    • 文件类型
      • 普通文件
        • 包含任意数据
        • 文本文件 ASCII && Unicode only
        • 二进制文件 else
        • 对内核而言，以上两者没什么区别
      • 目录
        • 目录是什么
          • 相关文件组的索引
          • 包含一组链接，把文件名映射到文件
        • 目录至少含有两个条目
          • 点 指向自身的链接
          • 点点 指向上层的链接
      • 套接字
        • 用来与另一个进程进行跨网络通信的文件
      • 其他文件类型
    • 每个进程都有一个当前工作路径，就是pwd的
    • 路径名
      • 绝对路径
        • 以/开始
      • 相对路径
        • 以点点开始
        • 或者以点开始

  • 10.4 读和写
    • read and write
      • read函数
        • 应用程序使用read实现读
        • 表示复制最多n个字节到内存位置buf
        • 代码示例：用read和write对终端进行读写，主要还是看一下思想。这里并没哟处理写的返回值，有可能是封装函数没有返回值
        • 返回-1表示错误，返回0表示EOF，其他情况返回读的字节值
      • write函数
        • 应用程序用write实现写
        • 从内存位置buf复制最多n个字节到fd表示的当前文件位置
        • ret n, ret -1 when error
      • 通过Iseek函数，可以修改在当前文件中的位置，这不在我们讲述的范围内
    • 潜在的问题，为什么会发生返回时 不足值/短计数呢
    • 1.遇到EOF
      • 如果文件剩的比想读的少，去read,提前遇到了EOF,只能读并返回剩余的字数
    • 2.从终端读
      • 从终端是按照行读的
      • 所以只返回一行的字数
    • 3.读写socket
      • 因为网络延迟等等
    • 如果我们写磁盘文件，或者读磁盘文件没遇到EOF，那么永远不会发生短计数
    • 不足值的处理
      • 如果我们想让程序健壮
      • 我们需要反复调用read和write，直到所有字节传输完毕

  • 10.8 共享文件
    • 三个数据结构——在Linux内核中，三个用来表示打开文件的数据结构
      • 描述符表
        • 每个对应单个进程
        • 由进程打开的文件标识符索引
        • 表项指向文件表的表项
      • 文件表
        • 所有进程共享
        • 内容
          • 当前文件位置
          • 引用计数（指向它的描述符表表项数
          • 指向v-node表对应表项的指针
        • 如果引用计数为零，内核会删除这个表项
      • v-node表
        • 所有进程共享
        • 包含stat结构（元数据 中的大部分信息
      • 我们看三个图来理解他们的关系
        • 注意到：描述符表表项和打开文件表表项始终是一一对应的
        • 三个表的关系图
        • 一个进程通过不同的描述符两次打开同一文件，两个描述符表项，指向两个打开文件表表项，指向同一个v-node表表项
          • 这样可以有不同的文件指针，从不同的位置开始读写
          • 可以对文件有不同的权限
        • 还有一种情况，父子进程之间，两个描述符表，同一标识符指向同一打开文件表表项，指向一个v-node表表项

  • 10.10 标准IO
    • 标注I/O是C语言提供的，比Unix I/O更高级别的替代
    • 如fopen,fclose,用于读写字节的fread,fwrite,用于读写字符串的fgets,fputs,和scanf,printf
    • 把打开的文件模型简化为一个流,和进程打开的三个文件类型，C程序有三个流
      • stdin
      • stdout
      • stderr
    • 流是对文件描述符和流缓冲区的抽象
    • 案例：降低系统级IO的开销，因为系统级I/O开销较大
      • 大概是这个意思，我们如果反复用getc去读，在此前应该调用一次系统级的(read)，然后此后getc从缓冲区读，不用调用read
  • 10.11 我该使用哪些IO
    • 1.只要有可能就用标准IO，对于磁盘和终端设备是首选
    • 2.不要用scanf等用于读文本文件的读二进制文件
    • 3.用RIO读socket
    • 注意两点限制
      • 不要在输入函数后立刻跟输出函数
      • 不要在输出函数中立刻跟输出函数
      • 涉及到一些和流的全双工性质相关的问题
